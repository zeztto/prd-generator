# PRD(Product Requirements Document) 종합 조사 보고서

> PRD 생성기 앱 개발을 위한 사전 자료 조사

---

## 1. PRD 개요

### 1.1 PRD란?
PRD(Product Requirements Document, 제품 요구사항 문서)는 **특정 제품이 무엇(What)을 해야 하고, 왜(Why) 그것을 해야 하는지를 정의하는 문서**이다.

핵심 질문:
- **어떤 문제를 해결하는가?** (Problem Statement)
- **누구를 위한 것인가?** (Target User)
- **어떤 기능이 필요한가?** (Features & Requirements)
- **성공을 어떻게 측정하는가?** (Success Metrics)

> PRD는 "어떻게(How) 구현할 것인가"보다는 "무엇을(What) 만들 것인가"에 초점을 맞춘다.

### 1.2 PRD의 목적과 중요성

| 목적 | 설명 |
|------|------|
| 공유된 이해 형성 | 팀 전체가 제품의 방향과 범위에 대해 동일한 이해를 갖도록 함 |
| 범위 관리 | 무엇을 포함/제외할지 명확히 하여 스코프 크리프를 방지 |
| 의사결정 기준 제공 | 개발 과정에서 트레이드오프 상황의 판단 근거 |
| 이해관계자 정렬 | 경영진, 개발팀, 디자인팀, 마케팅팀 등의 기대를 조율 |
| 우선순위 설정 | 기능 간 우선순위를 명확히 하여 리소스 배분 최적화 |

### 1.3 관련 문서와의 차이

```
비즈니스 전략
    ↓
  BRD (왜 이 사업을 하는가?)
    ↓
  MRD (시장이 원하는 것은?)
    ↓
  PRD (무엇을 만들 것인가?)
    ↓
기능 명세서 / 유저 스토리 (어떻게 구현할 것인가?)
```

| 비교 | PRD | MRD | BRD | 기능 명세서 | 유저 스토리 |
|------|-----|-----|-----|-----------|------------|
| 초점 | What & Why | 시장 기회 | 비즈니스 목표 | How (구현) | 단일 기능 |
| 추상화 | 중간 | 높음 | 매우 높음 | 낮음 | 낮음 |
| 작성자 | PM | PMM | 사업부 | 엔지니어링 리드 | PM/PO |
| 대상 | 전체 팀 | 경영진 | 투자자 | 개발팀 | 개발팀 |

### 1.4 PRD의 진화

| 시대 | 특징 |
|------|------|
| **전통적 (1990-2000s)** | 수십~수백 페이지, 완벽한 사전 정의, 변경 어려움, Word/PDF |
| **Agile 전환기 (2000-2010s)** | 1~5페이지로 축소, 유저 스토리 중심, Living Document |
| **현대적 (2020s~)** | 1-페이지 PRD, Notion/Confluence 기반, 데이터 중심, AI 활용 |

### 1.5 작성자와 독자

**주 작성자:** Product Manager (PM)

**기여자:**
- UX/UI 디자이너: 리서치, 와이어프레임
- 엔지니어링 리드: 기술적 실현 가능성
- 데이터 분석가: 정량적 근거
- QA 리드: 인수 기준 검토

**독자별 관심사:**

| 독자 | 주로 보는 부분 |
|------|--------------|
| 개발자 | 기능/비기능 요구사항, 인수 기준 |
| 디자이너 | 페르소나, 사용자 여정, 와이어프레임 |
| QA | 인수 기준, 엣지 케이스 |
| 경영진 | 문제 정의, 비즈니스 목표, KPI |
| 마케팅 | 타겟 사용자, 핵심 가치, 출시 일정 |

## 2. PRD 필수 구성요소 (13개 섹션)

### 2.1 제품 개요 (Product Overview)
- 제품/기능의 한 줄 요약
- 핵심 문제 (Problem Statement)
- 전략적 맥락
- 문서 버전 이력, 소유자

**좋은 예시:** "모바일 결제 앱 'QuickPay'는 소상공인이 별도 POS 단말기 없이 스마트폰만으로 카드 결제를 받을 수 있게 한다. 현재 소상공인의 43%가 카드 결제 인프라 부족으로 매출 기회를 잃고 있으며, Q3까지 MAU 50,000명 달성을 목표로 한다."

**나쁜 예시:** "결제 앱을 만든다. 여러 기능이 있을 예정이다."

### 2.2 목적과 목표 (Objectives & Goals)
- 비즈니스 목표, 사용자 목표
- 회사 전략과의 정렬
- 우선순위가 매겨진 목표 목록 (P0/P1/P2)
- 목표는 반드시 측정 가능(Measurable)해야 함

### 2.3 사용자/타겟 (Target Users / Personas)
- 핵심 페르소나 (2-3개 권장)
- 인구통계, 행동 패턴, 목표, 고충점, 기술 숙련도
- Jobs-to-be-Done (JTBD) 프레임워크

### 2.4 유저 스토리 (User Stories)
- 형식: "As a [역할], I want to [행동] so that [이유/혜택]"
- 인수 기준 (Acceptance Criteria) 필수
- 우선순위 태깅
- 페르소나 매핑

### 2.5 기능 요구사항 (Functional Requirements)
- 기능별 상세 명세, 입력/출력 정의
- 비즈니스 로직 및 규칙
- 예외 처리 및 엣지 케이스
- MoSCoW 우선순위: Must-have / Should-have / Could-have / Won't-have
- 고유 ID 부여 (예: FR-001)

### 2.6 비기능 요구사항 (Non-Functional Requirements)

| 카테고리 | 예시 |
|---------|------|
| 성능 | API 응답 시간 200ms 이내 (95th percentile) |
| 확장성 | 동시 접속자 100,000명 처리 |
| 보안 | PCI-DSS 준수, AES-256 암호화 |
| 가용성 | 월간 99.9% (다운타임 43분 이내) |
| 호환성 | iOS 15+, Android 12+ |
| 접근성 | WCAG 2.1 AA 기준 |
| 규정 준수 | 개인정보보호법, GDPR |

### 2.7 성공 지표 (Success Metrics / KPIs)
- 핵심 KPI와 목표 수치
- 측정 방법, 도구, 주기
- 기준선(Baseline) 데이터
- Vanity Metrics 피하기

### 2.8 타임라인/마일스톤 (Timeline/Milestones)
- 주요 마일스톤과 예상 날짜
- 단계별 산출물
- Go/No-Go 결정 시점

### 2.9 범위 (Scope - In/Out)
- In-Scope: 이번 릴리스에 포함
- Out-of-Scope: 이번에 하지 않음
- Future Consideration: 향후 고려

> Out-of-Scope를 명시하는 것이 In-Scope만큼 중요하다

### 2.10 제약 조건 (Constraints)
- 기술적, 예산, 일정, 인력, 법적/규제, 플랫폼 제약

### 2.11 의존성 (Dependencies)
- 내부 팀, 외부 서비스/API, 데이터, 선행 작업

### 2.12 가정 사항 (Assumptions)
- 사용자 행동, 기술적, 시장/비즈니스 가정
- 각 가정의 리스크 수준

### 2.13 릴리스 기준 (Release Criteria)
- 기능 완성도, 품질 기준, 성능 기준
- 보안 감사, 법적 검토 통과 여부

## 3. 선택적 섹션

| 섹션 | 포함 시점 |
|------|----------|
| 와이어프레임/목업 | UI 변경이 크거나 새 사용자 플로우 도입 시 |
| 기술 아키텍처 | 새 시스템 설계 또는 기존 아키텍처 대변경 시 |
| 경쟁사 분석 | 새 시장 진입 또는 차별화가 핵심일 때 |
| 리스크 분석 | 복잡도 높거나 외부 의존성 많을 때 |
| 데이터 모델 | 데이터 구조 변경이 핵심일 때 |
| 마이그레이션 계획 | 기존 시스템/데이터 전환 필요 시 |
| A/B 테스트 계획 | 점진적 롤아웃이나 실험 필요 시 |
| 롤백 계획 | 고위험 변경 시 |
| 용어 사전 | 도메인 특화 용어 많거나 크로스팀 협업 시 |

## 4. PRD 작성 프로세스

### 4.1 작성 전 준비
1. 사용자 리서치 (인터뷰, JTBD 프레임워크)
2. 이해관계자 매핑 및 인터뷰 (영향력 vs 관심도 매트릭스)
3. 경쟁 분석
4. 데이터 분석

### 4.2 초안 작성
1. 문제 정의를 먼저 → 기능은 그 다음
2. 요구사항 계층 구조: 테마 → 에픽 → 유저 스토리
3. 모든 요구사항을 사용자 페인 포인트에 연결
4. MVP 요구사항에 집중

### 4.3 리뷰 및 피드백
- 이해관계자 리뷰 및 서명 문서화
- 버전 관리 시스템 활용
- 반복적 업데이트

### 4.4 최종 확정 및 유지보수
- Living Document로 취급
- 프로젝트 관리 도구와 연동
- 스프린트마다 검토 및 수정

## 5. 베스트 프랙티스

### 5.1 해야 할 것
- **사용자 중심** 작성: 모든 요구사항을 사용자 pain point에 연결
- **측정 가능** 표현: 모호한 표현 대신 정량적 기준
- **Out of Scope 명시**: 문서화되지 않은 가정은 스코프 크립의 기회
- **시각 자료 활용**: 와이어프레임, 플로우차트 포함
- **우선순위 명확화**: MoSCoW 또는 RICE 활용

### 5.2 하지 말아야 할 것 (안티패턴)
- ❌ 문제 정의 전에 기능부터 작성
- ❌ 모호한 퍼소나 ("소상공인" → 구체적 인물상 필요)
- ❌ 성공 지표 누락
- ❌ 구현 방식(How) 지정 (UI 세부사항을 PRD에 넣기)
- ❌ "seamless", "intuitive" 같은 모호한 용어 사용
- ❌ 작성 후 방치 (Living Document로 관리해야)
- ❌ 이해관계자 입력 없이 혼자 작성

### 5.3 우선순위 설정 프레임워크

| 프레임워크 | 설명 | 적합 상황 |
|------------|------|-----------|
| **MoSCoW** | Must/Should/Could/Won't | 빠른 스프린트 결정 |
| **RICE** | Reach × Impact × Confidence / Effort | 분기별 로드맵 |
| **Kano** | 기본/성과/흥분 요소 분류 | 고객 만족도 분석 |

## 6. 주요 기업별 PRD 프레임워크

### 6.1 Amazon - Working Backwards (PR/FAQ)
- 가상의 보도자료(1페이지)와 FAQ(2페이지) 먼저 작성
- 고객 가치에서 역방향으로 사고
- Kindle, AWS, Alexa 등 탄생 배경

### 6.2 Intercom - Intermission
- A4 1페이지로 반드시 제한
- 문제 정의 + Job Stories + 성공 기준 + 범위
- 솔루션은 Figma에서 별도 진행
- RICE 프레임워크 활용

### 6.3 Kevin Yien (Square) PRD
- 5단계 워크플로우 기반
- 솔루션을 "경계선(Perimeter)"으로 그려 팀이 채우도록
- Problem → Goals/Non-Goals → Key Flows → Solution → Launch Plan

### 6.4 Shape Up (Basecamp) Pitch
- 5요소: Problem, Appetite(시간 제약), Solution, Rabbit Holes, No-Goes
- 상세 스펙 거부, 적절한 추상화 유지

### 6.5 Atlassian PRD
- 업계 "골드 스탠다드"
- Confluence 기반, Jira 연동
- 포괄적 체크리스트 제공

### 6.6 프레임워크 비교

| 프레임워크 | 분량 | 핵심 철학 | 적합 상황 |
|------------|------|-----------|-----------|
| Amazon PR/FAQ | 2페이지 | 고객 가치 역산 | 신규 제품/서비스 |
| Intercom | A4 1페이지 | 문제 정의 극한 | 기능 개선 |
| Kevin Yien | 6-8페이지 | 경계 그리기 | 범용 |
| Shape Up | 유동적 | 시간 제약 중심 | 시간 제약 프로젝트 |
| Atlassian | 상세 | 포괄적 문서화 | 대규모 팀 |

## 7. 유형별 PRD 차이

### 7.1 신규 제품 vs 기능 추가

| 구분 | 신규 제품 PRD | 기능 추가 PRD |
|------|-------------|-------------|
| 초점 | 시장 기회, 포지셔닝, 장기 비전 | 기존 제품 내 구현 |
| 분량 | 6-8페이지 | 1-3페이지 |
| 필수 섹션 | GTM 전략, 경쟁 포지셔닝 포함 | 간소화 가능 |

### 7.2 B2B vs B2C

| 구분 | B2B PRD | B2C PRD |
|------|---------|---------|
| 사용자 | 전문가, 제한된 수 | 일반 소비자, 대규모 |
| UX | 기능 복잡도 > 디자인 | 디자인 중요도 매우 높음 |
| 의사결정 | 다수 이해관계자, 긴 결정 주기 | 개인 결정, 감정/편의 기반 |

### 7.3 플랫폼별 특이사항

| 플랫폼 | 추가 고려사항 |
|--------|-------------|
| 모바일 앱 | 디바이스 제약, 오프라인 지원, 푸시 알림, 앱스토어 가이드라인 |
| 웹 서비스 | 브라우저 호환성, 반응형, SEO, 로딩 성능 |
| API/플랫폼 | 엔드포인트 명세, 인증/인가, Rate limiting, 버전 관리 |

## 8. 한국 시장 현황

### 8.1 PRD 사용 현황
- 2018년 이후 PM/PO 직군 확대와 함께 PRD 작성이 핵심 역량으로 자리잡음
- "기획서", "요구사항 정의서", "Product Spec" 등 다양한 용어 혼용
- Notion이 PRD 작성 도구로 압도적 인기

### 8.2 기업별 현황
- **대기업 (네이버, 카카오, 쿠팡, 토스):** 자체 PRD 템플릿, Confluence/Notion 활용
- **스타트업:** Notion 기반 경량 PRD, 1-2페이지
- **에이전시/SI:** 전통적 "요구사항 정의서"가 PRD 역할

### 8.3 한국어 PRD 도구 현황
- **한국어 전용 AI PRD 도구**: Vibe PRD Generator(vibegen.net)가 거의 유일
- 대부분 ChatGPT/Claude를 직접 활용
- 한국어에 최적화된 전문 PRD 생성 도구는 사실상 부재

## 9. 경쟁 분석 및 기회

### 9.1 주요 경쟁 도구

| 도구 | 특징 | 가격 |
|------|------|------|
| ChatPRD | 시장 1위, 10만+ PM 사용 | $8-24/월 |
| Figma Make | PRD AI 생성 | Figma 구독 내 |
| Miro AI | 비주얼 PRD | Free 10크레딧/월 |
| Productboard Spark | PRD + 로드맵 | $19/월 |
| Notion AI | 기존 워크스페이스 통합 | $10/유저/월 |

### 9.2 기회 영역 (블루오션)

1. **한국어 네이티브** PRD 도구 부재
2. **"배우면서 쓰는" 교육형** PRD 작성 도구 전무
3. 초보자 관점 진입장벽: 좋은 PRD 기준 부재, 품질 불안감, 구조 설계 어려움
4. 바이브 코딩 트렌드와 연계 (PRD → 코드 파이프라인)
5. 주니어 PM/학생 대상 합리적 가격

### 9.3 차별화 포인트 제안
