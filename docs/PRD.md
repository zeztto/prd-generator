## 1. 개요

### 1.1 제품 요약

| 항목 | 내용 |
|------|------|
| **제품명** | PRD 생성기 (PRD Generator) |
| **한 줄 설명** | PRD 작성 경험이 없는 한국어 사용자도 질문에 답하는 것만으로 전문가 수준의 PRD를 30분 내에 완성할 수 있는 AI 기반 웹 앱 |
| **타겟** | 한국어 사용자 — 개인(주니어 PM, 창업자, 디자이너) + 팀(스타트업, 기업 제품팀) |
| **핵심 가치** | "빈 페이지 공포증" 해결 — 배우면서 쓰는 교육형 PRD 작성 도구 |
| **플랫폼** | 반응형 웹앱 (데스크톱 + 모바일 최적화) + PWA |
| **사용 단위** | 개인 워크스페이스 (기본) + 팀 워크스페이스 (조직 단위) |

### 1.2 제품 비전

```
"PRD를 처음 써보는 사람도,
 질문에 답하다 보면 전문가 수준의 PRD가 완성되는 경험"
```

## 2. 배경 및 문제 정의

### 2.1 시장 배경

- 2018년 이후 한국 IT 업계에서 PM/PO 직군 급격히 확대
- PRD 작성이 PM 핵심 역량으로 자리잡았으나, 체계적 교육 부족
- 한국어 전용 PRD 도구가 사실상 부재 (Vibe PRD Generator 정도)
- 대부분 ChatGPT/Claude를 직접 활용하거나 Notion에 자유 형식으로 작성

### 2.2 핵심 문제

| # | 문제 | 영향 |
|---|------|------|
| 1 | **빈 페이지 공포증** — PRD를 시작할 때 무엇부터 써야 할지 모름 | 작성 자체를 포기하거나, 핵심 섹션을 누락한 불완전한 문서 생산 |
| 2 | **품질 기준 부재** — 좋은 PRD와 나쁜 PRD를 구분할 줄 모름 | 모호한 표현, 측정 불가능한 목표, 일관성 없는 내용 |
| 3 | **구조 설계 어려움** — 어떤 섹션이 필수인지, 어떤 순서로 작성해야 하는지 모름 | 비효율적인 문서 구조, 중복/누락 |
| 4 | **한국어 도구 부재** — 영어 기반 도구는 한국 IT 업계 용어/문화와 맞지 않음 | 도구 사용 포기, 수동 작성으로 회귀 |

### 2.3 기존 대안의 한계

| 대안 | 한계 |
|------|------|
| ChatGPT/Claude 직접 사용 | 매번 프롬프트 작성 필요, 일관성 없음, 품질 체크 부재 |
| Notion 자유 양식 | 가이드 없음, 섹션 누락 빈번, 품질 편차 큼 |
| ChatPRD (영문) | 한국어 미지원, 한국 업계 문맥 부재, 월 $8-24 |
| Google Docs 템플릿 | 정적 문서, AI 지원 없음, 각 항목 작성법 안내 없음 |

## 3. 목표 및 성공 지표

### 3.1 비즈니스 목표

| 목표 | 측정 기준 | 기간 |
|------|-----------|------|
| 한국어 PRD 도구 시장 1위 확보 | MAU 기준 | 12개월 |
| PRD 작성 진입장벽 제거 | PRD 완성률 50% 이상 | 6개월 |
| 교육형 UX로 차별화 | NPS 40+ | 6개월 |

### 3.2 사용자 목표

| 목표 | 기대 효과 |
|------|-----------|
| PRD 작성 시간 단축 | 기존 2-3시간 → 30-40분 |
| PRD 품질 향상 | 모호한 표현 감소, 필수 섹션 완성도 향상 |
| PRD 작성 역량 성장 | 사용할수록 가이드 의존도 감소 |

### 3.3 핵심 성공 지표 (KPIs)

| 메트릭 | 정의 | 목표 (3개월) | 목표 (12개월) | 측정 방법 |
|--------|------|-------------|-------------|-----------|
| 회원 가입 수 | 신규 가입 사용자 | 500명 | 5,000명 | DB 집계 |
| 활성화율 | 가입 후 PRD 1건 이상 완성 비율 | 40% | 55% | 퍼널 분석 |
| PRD 완성률 | 시작한 PRD 중 내보내기까지 도달 비율 | 35% | 55% | 이벤트 추적 |
| 평균 작성 시간 | 시작 → 내보내기 소요 시간 | 45분 | 30분 | 타임스탬프 |
| 품질 점수 평균 | AI 피드백 기준 | 65점 | 75점 | 시스템 산출 |
| D7 리텐션 | 7일 후 재방문 비율 | 25% | 35% | 코호트 분석 |
| PRD 당 AI 비용 | API 호출 비용 | $0.20 이하 | $0.15 이하 | 로그 집계 |
| NPS | 순추천지수 | 30+ | 40+ | 인앱 서베이 |

### 3.4 비목표 (Non-Goals)

- V1에서 **영어 등 다국어 지원하지 않는다**
- V1에서 **실시간 동시 편집은 지원하지 않는다** (팀 공유는 지원하되, 동시 편집은 아님)
- V1에서 **Jira/Linear 등 프로젝트 관리 도구 연동하지 않는다**
- V1에서 **유료 과금 시스템은 구현하지 않는다** (팀 기능 포함 전체 무료)
- **범용 문서 에디터가 되려 하지 않는다** — PRD 전용 도구로 집중
- **네이티브 모바일 앱은 만들지 않는다** — PWA로 모바일 경험 제공

## 4. 사용자 및 시나리오

### 4.1 타겟 페르소나

#### 페르소나 1: 주니어 PM 김지현 (1순위)

| 항목 | 내용 |
|------|------|
| 나이/직급 | 28세, PM 1년차 |
| 상황 | 스타트업에서 첫 PM 역할, PRD 작성 경험 없음 |
| 핵심 니즈 | PRD를 어떻게 쓰는지 배우면서 실제 문서를 완성하고 싶음 |
| 페인 포인트 | 빈 Notion 페이지 앞에서 2시간째 고민, 선배 PM의 피드백이 "구체적이지 않다" |
| 현재 해결 방법 | ChatGPT에 "PRD 써줘" → 결과물이 너무 일반적 → 결국 수동 수정 |

#### 페르소나 2: 1인 창업자 박서준 (2순위)

| 항목 | 내용 |
|------|------|
| 나이/직급 | 33세, 기술 창업자 |
| 상황 | MVP 개발 전 제품 방향을 정리해야 함, 개발 외주 의뢰용 문서 필요 |
| 핵심 니즈 | 머릿속 아이디어를 구조화된 문서로 빠르게 정리 |
| 페인 포인트 | 기획 전문가가 아님, 개발은 할 수 있지만 "무엇을 만들지" 정리가 어려움 |
| 현재 해결 방법 | 노션에 자유롭게 아이디어 나열 → 개발 착수 후 방향 수정 빈번 |

#### 페르소나 3: 커리어 전환 디자이너 이수진 (3순위)

| 항목 | 내용 |
|------|------|
| 나이/직급 | 30세, UX 디자이너 → PM 전환 준비 |
| 상황 | PM 포트폴리오용 PRD 작성 필요 |
| 핵심 니즈 | 전문적인 PRD 작성 능력을 증명 |
| 페인 포인트 | PRD 구조와 업계 표준을 모름, 면접에서 요구하는 수준이 불명확 |
| 현재 해결 방법 | 블로그에서 PRD 예시 검색 → 따라 쓰기 → 자기 프로젝트에 맞는지 확신 없음 |

#### 페르소나 4: 스타트업 제품팀 리드 최민호 (2순위 — 팀 사용)

| 항목 | 내용 |
|------|------|
| 나이/직급 | 35세, Head of Product |
| 상황 | 5명의 PM이 있는 팀을 이끌며, PRD 작성 표준화와 품질 관리가 필요 |
| 핵심 니즈 | 팀 전체가 일관된 형식의 PRD를 작성하고, 작성 현황을 한눈에 파악 |
| 페인 포인트 | PM마다 PRD 형식이 달라 리뷰가 비효율적, 주니어 PM의 PRD 품질 관리 어려움 |
| 현재 해결 방법 | Notion 템플릿 공유 → 개인마다 변형하여 사용 → 일관성 붕괴 |

#### 페르소나 5: 시스템 관리자 (관리자 대시보드)

| 항목 | 내용 |
|------|------|
| 역할 | Super Admin |
| 핵심 니즈 | 서비스 전체 사용 현황, AI 비용, 사용자 관리, 시스템 건강 모니터링 |

### 4.2 핵심 사용자 시나리오

**시나리오 1: 주니어 PM 김지현의 첫 PRD**

> 월요일 아침, 팀 리드로부터 "이번 분기에 추가할 실시간 채팅 기능 PRD를 작성해 달라"는 요청을 받은 지현. PRD 생성기에 접속하여 "기존 기능 개선" 유형을 선택하고, "중고 거래 앱에 실시간 채팅 기능 추가"라는 한 줄 설명을 입력한다. 시스템이 안내하는 질문에 하나씩 답하면서, 각 섹션의 가이드와 좋은 예시를 참고한다. 모호하게 작성한 "많은 사용자가 불편해한다"는 문장에 AI가 "구체적 수치로 바꿔보세요"라고 제안하고, 수정한다. 40분 후 품질 점수 82점의 PRD가 완성되어 Notion으로 내보낸다.

**시나리오 2: 창업자 박서준의 MVP 정리**

> 아이디어 노트를 열어놓고 PRD 생성기에 접속한 서준. "신규 제품" 유형을 선택하고, 핵심 아이디어를 키워드로 나열한다. AI가 이를 구조화된 문단으로 변환해주고, 경쟁사 분석과 KPI까지 제안한다. 30분 만에 외주 개발사에 전달할 수 있는 수준의 PRD가 완성된다.

**시나리오 3: 팀 리드 최민호의 팀 온보딩**

> 최민호는 PRD 생성기에 팀 워크스페이스를 생성하고, 5명의 PM에게 초대 링크를 전달한다. 팀 전용 PRD 템플릿을 설정하여 모든 팀원이 동일한 형식으로 PRD를 작성하도록 한다. 대시보드에서 팀원들의 PRD 작성 현황과 품질 점수를 한눈에 확인하고, 특정 PRD에 코멘트를 남긴다.

**시나리오 4: 출퇴근길 모바일 PRD 작성**

> 지하철에서 주니어 PM 김지현이 스마트폰으로 PRD 생성기를 연다. PWA로 설치되어 있어 앱처럼 바로 실행된다. 모바일에 최적화된 단일 컬럼 레이아웃에서 어제 작성하던 PRD를 이어서 작업한다. 키워드만 빠르게 입력하고 AI가 문장으로 확장해준다. 사무실 도착 후 데스크톱에서 동일한 PRD를 열어 3컬럼 뷰로 상세 편집을 이어간다.

## 5. 사용자 워크플로우

### 5.1 전체 흐름

```
[홈 화면]
    │
    ├─ 새 PRD 작성하기
    │       │
    │       ▼
    │   [STEP 1: 프로젝트 설정] ─── 이름, 유형, 플랫폼, 타겟
    │       │
    │       ▼ (분기 로직 적용)
    │
    │   [STEP 2: 단계별 PRD 작성] ─── 5-7개 서브 스텝
    │       │
    │       ├─ 2-1. 배경/문제 정의 ──── AI 보강 가능
    │       ├─ 2-2. 목표/성공 지표 ──── AI 지표 제안
    │       ├─ 2-3. 타겟 사용자 ─────── AI 페르소나 생성
    │       ├─ 2-4. 핵심 기능 ────────── AI 기능 제안
    │       ├─ 2-5. 사용자 시나리오 ─── AI 시나리오 생성
    │       ├─ 2-6. 기술 요구사항 ───── (유형별 선택)
    │       └─ 2-7. 일정/마일스톤 ───── (선택)
    │       │
    │       ▼
    │
    │   [STEP 3: AI 검토 및 보강]
    │       │
    │       ├─ 빠진 섹션 탐지
    │       ├─ 모호한 서술 개선 제안
    │       ├─ 일관성 검사
    │       └─ 완성도 점수 부여
    │       │
    │       ▼
    │
    │   [STEP 4: 미리보기 및 최종 편집]
    │       │
    │       ├─ 실제 문서 형태 렌더링
    │       ├─ 인라인 편집
    │       └─ 섹션 순서 변경
    │       │
    │       ▼
    │
    │   [STEP 5: 내보내기]
    │       │
    │       ├─ Markdown
    │       ├─ PDF
    │       ├─ Notion 복사
    │       └─ 공유 링크
    │
    └─ 이어서 작성하기 ──→ (마지막 편집 위치로 이동)
```

### 5.5 AI 어시스턴트 인터랙션

| 시점 | AI 동작 | 트리거 |
|------|---------|--------|
| 입력 후 | 품질 평가 + 개선 제안 | "AI로 보강하기" 버튼 클릭 |
| 섹션 완료 시 | 다음 섹션과의 일관성 검사 | 자동 (백그라운드) |
| 전체 완료 시 | 종합 검토 + 빠진 부분 탐지 | STEP 3 진입 시 자동 |
| 언제든 | 자유 질문 응답 | "AI에게 물어보기" 클릭 |

**AI 보강 3가지 모드:**
1. **구조화**: 자유 서술을 구조적 형태로 변환
2. **구체화**: 모호한 표현을 구체적으로 변환 (추정 데이터는 명확히 표시)
3. **확장**: 빠진 관점을 추가로 제안

**AI 제안 수락 UI:**
```
 원본: "채팅 기능이 없어서 불편함"
     ↓ [AI로 보강하기]
 AI 제안: "구매자-판매자 간 실시간 소통 수단 부재로 인해
          거래 완료율이 저하되고 있으며..."

 [✅ 수락]  [✏️ 수정 후 수락]  [↩️ 원본 유지]
```

### 5.6 에러/엣지 케이스 처리

| 상황 | 처리 방식 |
|------|-----------|
| 중간 이탈 | 2초 debounce 자동 저장, 재방문 시 "이어서 작성하기" |
| 필수 정보 누락 | 해당 필드 하이라이트 + 구체적 안내 메시지 |
| 모호한 입력 | AI 클래리파잉 질문 (선택지 포함) |
| AI 호출 실패 | "다시 시도" + 직접 작성 계속 가능 |
| 극단적 입력 (5,000자+) | "핵심만 요약하거나 섹션 분리" 안내 |
| 다중 탭 충돌 | "최신 버전 불러오기" 알림 |

#### FR-001: 단계별 PRD 생성 워크플로우

## 7. 인증 및 사용자 시스템

> 상세 설계: [DETAILED-DESIGN.md §1](./DETAILED-DESIGN.md#1-인증-시스템-상세-설계)

### 7.1 가입 방식

| 방식 | 프로바이더 | OAuth Scope |
|------|-----------|-------------|
| 소셜 로그인 | Google | `openid email profile` |
| 소셜 로그인 | GitHub | `read:user user:email` |
| 소셜 로그인 | Kakao | `profile_nickname profile_image account_email` |
| 이메일/패스워드 | 자체 | bcrypt 해싱, 이메일 인증 (Resend API) |

### 7.2 가입 플로우

```
[랜딩 페이지]
    ↓
[로그인/회원가입 페이지]
    ├── 소셜 로그인 → OAuth → 신규면 [온보딩] → [대시보드]
    │                         기존이면 → [대시보드]
    └── 이메일 가입 → [가입 폼] → [이메일 인증] → [온보딩] → [대시보드]

[온보딩] (소셜/이메일 공통, 최초 1회):
    ├── Step 1: 약관 동의 (필수: 이용약관, 개인정보처리방침 / 선택: 마케팅)
    └── Step 2: 추가 프로필 (선택: 직군, 경력)
```

### 7.3 인증 아키텍처

| 항목 | 선택 | 근거 |
|------|------|------|
| 프레임워크 | NextAuth.js v5 | Next.js 네이티브 통합 |
| 세션 전략 | **DB 세션** (JWT 아님) | 즉시 무효화 가능, 팀 권한 변경 즉시 반영 |
| 세션 TTL | 30일 (슬라이딩 윈도우) | 활동 시마다 갱신 |
| 비밀번호 | bcrypt (salt rounds: 12) | 업계 표준 |
| 이메일 인증 | 30분 TTL 토큰, 인증 전 PRD 3건 제한 | 스팸 방지 + 허들 최소화 |

### 7.4 사용자 프로필

| 필드 | 필수 | 수정 가능 |
|------|:----:|:---------:|
| 이름 | ✅ | ✅ |
| 이메일 | ✅ | ✅ (재인증 필요) |
| 아바타 | ❌ | ✅ (업로드 또는 소셜 프로필) |
| 직군 | ❌ | ✅ |
| 경력 | ❌ | ✅ |
| 소셜 계정 연결 | - | ✅ (추가/해제, 최소 1개 유지) |

### 7.5 계정 삭제

- 30일 유예 기간 (기간 내 로그인 시 복구 가능)
- 유예 후 데이터 익명화 (개인 PRD 삭제, 팀 PRD는 소유권 이전)
- GDPR/개인정보보호법 준수: JSON 데이터 내보내기 제공

| PRD 삭제 | ✅ | ✅ | 자기 것만 | ❌ |
| PRD 읽기 | ✅ | ✅ | ✅ | ✅ |
| 템플릿 관리 | ✅ | ✅ | ❌ | ❌ |
| 활동 로그 조회 | ✅ | ✅ | ❌ | ❌ |

### 8.3 멤버 초대

| 방식 | 설명 | 만료 |
|------|------|------|
| 이메일 초대 | 특정 이메일로 초대 링크 전송 | 7일 |
| 초대 링크 | 누구나 참여 가능한 링크 생성 | 설정 가능 (1일/7일/30일/무기한) |

### 8.4 워크스페이스 전환 UI

```
┌─ 사이드바 상단 ──────────────┐
│ ▾ 김지현의 워크스페이스       │  ← 드롭다운 클릭
│   ┌────────────────────────┐ │
│   │ ● 개인 워크스페이스     │ │
│   │ ──────────────────     │ │
│   │ ○ 스타트업A 팀         │ │
│   │ ○ 프로젝트B 팀         │ │
│   │ ──────────────────     │ │
│   │ + 새 팀 만들기         │ │
│   └────────────────────────┘ │
└──────────────────────────────┘
```

- 전환 시 해당 워크스페이스의 PRD 목록으로 즉시 이동
- 현재 워크스페이스 표시 (사이드바 상단에 항상 표시)

### 8.5 팀 PRD 관리

- 조직 내 모든 PRD는 멤버에게 자동 공유 (권한에 따라 읽기/편집)
- PRD 소유권 이전: 개인 → 팀 (복사 또는 이동 선택), 팀 → 개인 (Admin 이상)
- 팀 활동 로그: PRD 생성/편집/삭제/내보내기 기록 (90일 보관)

## 10. PRD 섹션 구조 및 질문 설계

### 7.1 표준 PRD 구조 (10개 섹션)

| # | 섹션 | 목적 |
|---|------|------|
| 1 | 개요 | 프로젝트 핵심을 한눈에 파악 |
| 2 | 배경 및 문제 정의 | 왜 이 프로젝트를 하는지 맥락 제공 |
| 3 | 목표 및 성공 지표 | 성공의 기준 명확 정의 |
| 4 | 사용자 및 시나리오 | 누구를 위한 것인지 구체화 |
| 5 | 요구사항 | 무엇을 만들어야 하는지 정의 |
| 6 | 솔루션 개요 | 어떻게 문제를 해결할지 방향 제시 |
| 7 | 범위 및 마일스톤 | In/Out of Scope, 일정 |
| 8 | 리스크 및 의존성 | 실패 가능성 사전 인식 |
| 9 | 부록 | 보충 자료 (경쟁사, 리서치, 용어) |
| 10 | 검토 및 승인 | 리뷰어, 변경 이력, 미결 질문 |

### 7.2 질문 설계 요약 (총 30개)

#### 단계별 질문 배치

| 단계 | 화면 제목 | 질문 수 | 예상 소요 | 핵심 |
|------|-----------|---------|-----------|------|
| Step 1 | "시작해볼까요?" | 4개 | 2-3분 | 쉬운 질문으로 진입장벽 ↓ |
| Step 2 | "어떤 문제를 풀려고 하나요?" | 4개 | 5-7분 | 핵심 사고 구간, AI 보조 활발 |
| Step 3 | "성공은 어떤 모습인가요?" | 7개 | 7-10분 | 가장 어려운 구간 |
| Step 4 | "무엇을 만들어야 하나요?" | 7개 | 7-10분 | AI 초안 생성 적극 활용 |
| Step 5 | "언제까지, 어떻게?" | 6개 | 5-7분 | 실행 계획, 현실감 |
| Step 6 | "마무리!" | 6개 | 3-5분 | 선택 항목, 빠른 완료감 |

**총 예상 시간: 30-40분 (전체) / 15-20분 (필수만)**

#### 질문 설계 핵심 원칙

모든 질문에 포함되는 요소:
- **질문 텍스트**: 친근하고 이해하기 쉬운 한국어 톤
- **가이드 텍스트**: 왜 이 정보가 필요한지 1-2줄 설명
- **좋은 예시 / 나쁜 예시**: 구체적인 작성 방향 제시
- **AI 보조 여부**: AI가 초안을 생성해줄 수 있는지 표시
- **필수 여부**: 스킵 가능/불가능 구분

#### 대표 질문 예시

**Q: "해결하려는 핵심 문제가 무엇인가요?"**
- 유형: 주관식 (장문)
- 가이드: "문제를 구체적으로 정의할수록 솔루션도 명확해집니다."
- ✅ 좋은 예시: "모바일 결제 과정에서 주소 입력 단계의 이탈률이 62%에 달한다. 사용자 인터뷰에서 10명 중 7명이 '입력칸이 너무 많다'고 응답했다."
- ❌ 나쁜 예시: "결제가 불편함"
- AI 보조: 예 (문제 구조화 - 상황→원인→결과 프레임)

### 7.3 프레임워크별 질문 변형

| 프레임워크 | 총 질문 수 | 핵심 변경점 | 예상 시간 |
|------------|-----------|------------|-----------|
| **표준 PRD** | 30개 | 기본 전체 구조 | 30-40분 |
| **Amazon PR/FAQ** | 24개 | 보도자료 헤드라인 + 고객/내부 FAQ 추가 | 25-35분 |
| **1-Pager** | 10개 | 극단적 간결화, 핵심만 | 10-15분 |

### 7.4 품질 체크 시스템

#### 자동 감지 항목

**모호한 표현 감지 (8패턴):**

| 감지 패턴 | 피드백 |
|-----------|--------|
| "빠르게", "빨리" | "'빠르게'가 구체적으로 어느 정도인가요? 예: '500ms 이내'" |
| "많은", "대부분" | "'많은'을 숫자로 표현해보세요. 예: '사용자의 60%'" |
| "좋은", "개선된" | "어떤 면에서, 얼마나 좋아지나요? 현재 값과 목표 값을 적어보세요." |
| "쉬운", "직관적인" | "'쉬운'의 기준이 뭔가요? 예: '3번의 탭으로 완료'" |
| "등", "기타" | "구체적으로 어떤 것들이 포함되나요? 가능하면 모두 나열해주세요." |
| "적절한", "합리적인" | "어느 수준인지 구체화해주세요. 숫자나 조건을 함께 적어보세요." |
| "최대한", "가능한 한" | "명확한 기준이 필요해요. 예: '99.9% 가동률'" |
| "~할 수도 있다" | "포함하나요, 안 하나요? 결정 안 됐으면 '미결 질문'으로 옮겨주세요." |

**일관성 검사 (7항목):**
- 목표 ↔ KPI 정합성
- 문제 ↔ 솔루션 정합성

- 용어 일관성
- 비목표 ↔ 제외 범위 정합성
- 일정 순서 논리

## 11. AI 시스템 설계

### 8.1 LLM 선택

| 용도 | 모델 | 이유 |
|------|------|------|
| 섹션 자동 생성 | Claude Sonnet | 비용-품질 균형, 한국어 품질 우수 |
| 품질 피드백/검토 | Claude Sonnet | 분석력 필요 |
| 보완 질문/간단 제안 | Claude Haiku | 빠른 응답, 저비용 |
| Fallback | GPT-4o-mini | Claude 장애 시 |

### 8.2 프롬프트 엔지니어링 전략

**시스템 프롬프트 핵심:**
```
역할: 경험 많은 한국 IT 기업의 시니어 PM
출력 규칙:
- 모든 출력 한국어
- 비즈니스/기술 용어는 한국어+영어 혼용 (예: "사용자 스토리(User Story)")
- 구체적이고 측정 가능한 표현
- "적절한", "다양한" 같은 모호 표현 금지
포맷: Markdown
```

**컨텍스트 조립 흐름:**
```
시스템 프롬프트 (역할 + 규칙)
    + 템플릿 가이드 (선택된 프레임워크)
    + 이전 섹션 요약 (일관성 유지)
    + 현재 섹션 지시사항 + 사용자 입력
        → LLM API 호출 (스트리밍)
            → 출력 후처리 (파싱, 검증)
                → 사용자에게 스트리밍 표시
```

### 8.3 AI 품질 관리

- **Hallucination 방지**: 통계/시장 데이터 생성 시 "예시 수치" 임을 명시
- **일관성 유지**: 이전 섹션을 컨텍스트로 제공, 핵심 용어 목록 자동 추출
- **품질 게이트**: 최소 길이 검증, 필수 키워드 포함 여부, 반복/무의미 내용 탐지

### 8.4 토큰 사용량 (PRD 1건 기준)

| 작업 | 입력 토큰 | 출력 토큰 | 비용 (Sonnet) |
|------|-----------|-----------|---------------|
| 섹션 생성 (12개) | ~24,000 | ~12,000 | ~$0.12 |
| 품질 피드백 | ~8,000 | ~2,000 | ~$0.03 |
| 보완 질문 (5회) | ~5,000 | ~1,500 | ~$0.02 |
| **합계** | **~37,000** | **~15,500** | **~$0.17** |

| 태블릿 | 768-1023px | 2컬럼 | 가이드 패널 토글, 접이식 사이드바 |
| 데스크톱 | 1024px+ | 3컬럼 (사이드바240px + 메인 + 미리보기360px) | 키보드 단축키, 실시간 미리보기 패널 |

**모바일 PRD 작성 최적화:**
- 장문 입력 대신 키워드 태그 → AI가 문장으로 확장
- 입력 보조 칩 (자주 쓰는 표현 원탭 삽입)
- 가이드 패널은 바텀시트로 변환 (필요 시 위로 스와이프)
- 미리보기는 별도 탭으로 분리

**디바이스 간 동기화:**
- 모바일 작성 → 데스크톱 이어서 (DB 자동 동기화)
- 브레이크포인트 전환 시 스크롤 위치/입력 상태 유지

### 12.6 PWA (Progressive Web App)

> 상세 설계: [DETAILED-DESIGN.md §4](./DETAILED-DESIGN.md#4-pwa-설계)

| 항목 | 설명 |
|------|------|
| Manifest | `name`, `short_name`, `icons`(8종), `theme_color`, `shortcuts`, `screenshots` |
| Service Worker | Workbox 기반, 리소스별 6가지 캐싱 전략 |
| 오프라인 | 정적 셸 캐시 + 작성 중 PRD 읽기 가능 + 오프라인 폴백 페이지 |
| 설치 | 3회 방문 이후 A2HS 프롬프트, iOS/Android 스플래시 스크린 |
| 업데이트 | SW 업데이트 시 "새 버전 사용 가능" 토스트 → 새로고침 |

**캐싱 전략:**

| 리소스 | 전략 | 설명 |
|--------|------|------|
| App Shell (HTML/CSS/JS) | Precache | 빌드 시 캐시, 즉시 로딩 |
| API 응답 (/api/prd/*) | NetworkFirst | 네트워크 우선, 실패 시 캐시 |
| AI 응답 (/api/ai/*) | NetworkOnly | 항상 네트워크 (캐시 부적합) |
| 이미지/폰트 | CacheFirst | 캐시 우선, 없으면 네트워크 |
| 정적 에셋 | StaleWhileRevalidate | 캐시 즉시 반환 + 백그라운드 갱신 |

**오프라인에서 가능/불가능한 것:**

| 가능 | 불가능 |
|------|--------|
| 작성 중인 PRD 읽기 | 새 PRD 생성 (AI 필요) |
| 앱 탐색 (캐시된 페이지) | AI 섹션 생성/보강 |
| 오프라인 폴백 페이지 표시 | 내보내기 (PDF 서버 렌더링) |
| | 팀 기능 (실시간 동기화) |

## 15. 범위 및 마일스톤

### 15.1 범위

**In-Scope (V1):**
- 인증 시스템 (소셜 로그인 3종 + 이메일/패스워드 + 이메일 인증)

- AI 기반 섹션 자동 생성 + 품질 피드백
- 5종 기본 템플릿
- 내보내기 (Markdown, PDF)
- 자동 저장 (로컬 + DB 동기화)
- 관리자 대시보드 (사용자/PRD/AI 통계, 사용자 관리, 시스템 모니터링)
